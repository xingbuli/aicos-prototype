# AICOS

AICOS is an AI chief of staff that does work, not just tracks it: it turns objectives into roadmaps, prepares a leader's week, and drafts the follow-through. The product thesis is that useful AI for leaders needs a visible trust spine: draft, don't send; show confidence; name the gaps; respect the walls.

## For Evaluators

Live URL: https://aicos-prototype.vercel.app

Open the deployed URL, choose one of the three workspaces, confirm the pre-selected source setup, and take the guided tour. No account, environment variable, or real data is required. The intended first run is:

1. Choose Claudia, Alfredo, or Alex.
2. Confirm or adjust the simulated connected sources for that workspace.
3. Follow the 5-step tour.
4. Read the weekly briefing.
5. Edit a draft, approve it, edit again if needed, then mark it sent in the prototype.
6. Open Prep Desk to review, stage, and complete simulated deck prep, meeting prep, 1:1 agenda prep, nudge requests, and scheduling work.
7. Open chat as an in-page side panel without leaving the current view.
8. Generate a roadmap from your own objective.
9. Request access from a blind spot or off-limits source.

| Authentication | Simulated | Choosing a workspace is persona selection, not real auth. |
| Sending messages | Simulated | AICOS drafts messages and the leader can mark them sent in the prototype. No real message is sent. |

## Two-Month MVP Roadmap

Trust is earned by measurement, so measurement runs from Week 1 — not bolted on at the end.

**The feedback loop (three nested loops, live from Week 1):**

- **In-product (continuous).** Every confidence tag, blind-spot call, and inferred owner/date carries a one-tap **Right / Wrong / I'll fix**. Each tap is a labeled data point — the trust UI doubles as the accuracy-labeling pipeline — and every inference is logged with its source and the leader's eventual verdict.
- **Weekly design-partner review (qualitative).** A 30-minute session with each of Claudia, Alfredo, and Alex: what did AICOS get wrong, what did it miss, and did any flag feel like over-reporting (Alfredo's exact fear)? Captured as a correction log.
- **Biweekly metric review against gates (quantitative).** The numbers below decide whether the next phase's higher-stakes capability unlocks.

**Baseline first (Week 1).** Capture the starting point the interviews already quantified — admin load (Alex 8–16 hrs/wk, Claudia 20%+, Alfredo 15–20%) and blockers-missed/week — before changing anything.

**Success metrics**

- **Trust (the gate for higher-stakes capability):** briefing **precision** (% of flagged items the leader confirms real); **false-positive rate** (% of "stale/at-risk" flags that were actually fine — target → ~0); **blind-spot correctness** (% of blind-spot flags the leader agrees were genuinely unseen); **inference acceptance** (% of inferred owners/dates confirmed without edit).
- **Value:** self-reported admin hours saved vs. Week-1 baseline; weekly active leaders; draft-approval rate; % of staged Prep Desk actions committed; time-to-first-approved-action.
- **Safety (hard invariants, never traded off):** zero unauthorized sends/writes; zero restricted-category accesses.

**Weeks 1–3 — Real visibility (read-only).** Scoped read-only connectors for the tools these users actually use: Calendar + Email, Jira (Alex), Microsoft Loop (Claudia), Power BI (Alfredo), plus the team's chat (Slack or Teams). Permissions inherit what each user already has — no parallel access layer. *Gate to advance: baseline captured; precision and blind-spot correctness measured and trending up.*

**Weeks 3–5 — Trust hardening (and provider-neutral routing).** A staleness/coverage model behind the confidence tags, an audit log of every inference and its source (this *is* the measurement substrate the metric review consumes), human-in-the-loop gates on any outbound action, and the restricted-category wall enforced server-side. Model/provider selection lands here as infrastructure, not a separate milestone: a **routing agent** picks the model per task behind the same trust layer — high-stakes reasoning (blind-spot determination, roadmap synthesis) to the strongest model, cheap high-volume work (status classification, tagging) to fast/low-cost models, with vendor fallback — and the router's own decisions are measured (per-route accuracy and cost feed back so the policy improves). This operationalizes "model/provider selection is future work" without locking to one vendor. *Gate to unlock write-back: false-positive rate below threshold and inference acceptance above threshold, sustained two weeks across all three leaders.*

**Weeks 4–6 — Execution orchestration.** Two-way write-back behind approval (create/assign tasks in Loop/Jira), scheduled nudges and follow-ups, and a draft-deck reviewer that checks slides against a corporate style guide (Alfredo's ask). *Measured: action-approval rate and post-write correction rate (did a write need undoing?).*

**Weeks 6–7 — Proactive intelligence.** A weekly competitive/regulatory digest from a configurable watchlist (Alfredo lost his subscriptions in the reorg), and priority-shift detection across objectives.

**Weeks 7–8 — Adoption & accuracy.** Guided onboarding with a human account manager (Claudia's ask), a team mode (AICOS supports the whole team, not one leader), and an evaluation harness that formalizes the metrics tracked since Week 1 into a regression suite. "Accurate without excessive intervention" becomes a measured target: inference acceptance up, correction rate down.

Throughout: lowest-blast-radius first (read, then draft), earning higher-stakes capability (write, then act) only as accuracy is proven — and now "proven" is a number, not a feeling.
